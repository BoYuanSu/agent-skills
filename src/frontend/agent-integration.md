# AI Agent 整合與規格驅動開發 (Spec-Driven Development) 探討

本文件探討前端架構如何透過「規格驅動」與「分層解耦」的設計，與 AI Agent 進行高效整合與協作。

## 1. 核心設計概念：先定義兩端，延後在接縫處造橋

為了降低 AI Agent 在開發複雜功能時的重工率與邏輯耦合，我們採取**「定義兩端 Interface，延後實作 Adapter」**的開發策略：

*   **定義兩端（呈現與請求）：**
    *   **呈現層模組 (Presentation Module)：** 優先關注畫面的呈現與互動。不考慮後端 API 的實際結構，先根據 UI 設計逆推定義出「最理想、最易渲染」的**呈現介面 (Presentation Interface)**，並使用假資料 (Mock Data) 獨立完成呈現層模組的渲染與邏輯開發。
    *   **資料請求模組 (Request Module)：** 專注於處理使用者操作的查詢參數，發送請求以取得後端的原始資料（Raw Data）。
*   **在接縫 (Seam) 造橋（適配器）：**
    *   **正規化適配器 (Normalization Adapter)：** 位於兩端的接縫處。當呈現層模組（Mock 驗證過）與資料請求模組（順利取得原始資料）這兩端都穩固後，才在此處實作轉譯邏輯，將 API 原始資料轉換為呈現介面所預期的標準格式。

---

## 2. 規格驅動的協作模式 (AI Agent 協作流)

為了確保 AI Agent 產出的程式碼具備高維護性，將工作流拆分為**「分析規劃 (Spec/Task)」**與**「程式碼實作」**兩個階段。AI Agent 在進行任何實作前，必須先分析並產出明確的 Spec，經確認後才進行開發：

### 階段一：專案評估與架構定調
*   **評估複雜度：** 評估頁面是否複雜。
    *   *簡單頁面：* 捨棄正規化適配器，簡化為「資料請求模組 (Request Module) ➔ 呈現層模組 (Presentation Module)」。
    *   *複雜功能：* 確立完整的「資料請求模組 (Request Module) ➔ 正規化適配器 (Normalization Adapter) ➔ 呈現層模組 (Presentation Module)」三層架構。

### 階段二：呈現層先行（不阻塞開發）
1.  **呈現層規格分析（分析期）：** AI Agent 分析視圖切分與資料合約，**僅產出**呈現層模組的 spec/task（不進行任何實作）。
2.  **呈現層模組開發（實作期）：** 根據前一步的 spec/task，以 **呈現介面 (Presentation Interface)** 為核心，使用假資料進行呈現層元件的開發與測試。

### 階段三：資料請求層對接
3.  **資料請求模組實作：** 實作資料請求模組的參數收集與 API 發送，確認能順利取回後端最原始的資料結構（Raw Data）。

### 階段四：接縫整合（延後分析與實作）
4.  **適配器規格分析（分析期 - 延後執行）：** 當呈現層模組（Mock 驗證過）與資料請求模組（真實原始資料）皆就緒後，AI Agent 對比兩端資料結構的落差，**僅產出**正規化適配器 (Normalization Adapter) 的轉譯 spec/task。
5.  **適配器邏輯實作（實作期）：** 根據轉譯 spec/task，實作正規化適配器，將 API 原始資料轉譯為符合呈現介面的資料，在接縫處完成系統縫合。

### 階段五：狀態連動與邊界控制
6.  **狀態與控制實作：** 實作純視覺開關與全域事件偵測的狀態連動。

## 3. 延後「資料正規化」與統一術語的設計優勢

*   **避免無效的假設接縫 (Hypothetical Seam)：** 在開發初期，UI 與 API 的規格仍可能有變動。若過早設計與分析轉譯邏輯，極易因為規格調整而需要重複修改。延後造橋可確保接縫是基於「真實落差」而建立。
*   **極大化局部性 (Locality)：** 當 API 原始資料結構變更時，修改的範疇被完全限縮在正規化適配器（Adapter）內部，呈現層模組（Presentation Module）完全不需變動。
*   **優化測試表面積 (Test Surface)：** 呈現介面是呈現層模組唯一的測試表面積。藉由解耦，我們可以使用簡單的 Mock 資料完整測試 UI 渲染，而正規化適配器（純函數）也可以獨立進行單元測試，大幅提升整體測試性。
