# AI Agent 整合與規格驅動開發 (Spec-Driven Development) 探討

本文件用於探討前端架構如何透過「規格驅動」與「分層解耦」的設計，與 AI Agent 進行高效整合與協作。

## 1. 核心設計概念：先定義兩端，最後再造橋

為了降低 AI Agent 在開發複雜功能時的重工率與邏輯耦合，我們採取**「兩端先行，延後造橋」**的開發策略：

*   **定義兩端（UI 與 API）：**
    *   **呈現層 (UI/Mock)：** 優先關注畫面的呈現與互動。不考慮後端 API 的實際結構，先根據 UI 設計逆推定義出「最理想、最易渲染」的標準化視圖資料格式，並使用假資料 (Mock Data) 獨立完成 UI 元件的渲染與邏輯開發。
    *   **資料請求層 (API)：** 專注於處理使用者操作的查詢參數，發送請求以取得後端的原始資料。
*   **最後造橋（正規化層）：**
    *   **資料正規化層 (Normalization/Transform)：** 在 UI 呈現層（Mock 驗證過）與資料請求層（取得真實原始資料）這兩端都穩固後，才在此層實作轉譯邏輯，將 API 原始資料轉換為呈現層預期的標準化格式。

---

## 2. 規格驅動的協作模式 (AI Agent 協作流)

為了確保 AI Agent 產出的程式碼具備高維護性，將工作流拆分為**「分析規劃 (Spec/Task)」**與**「程式碼實作」**兩個階段。AI Agent 在進行任何實作前，必須先分析並產出明確的 Spec，經確認後才進行開發：

### 階段一：專案評估與架構定調
*   **評估複雜度：** 評估頁面是否複雜。
    *   *靜態/簡單頁面：* 捨棄正規化層，簡化為「請求 (API) -> 呈現 (UI)」。
    *   *複雜功能：* 確立完整的「參數/請求 -> 正規化 -> 呈現」架構。

### 階段二：UI 先行（不阻塞開發）
1.  **UI 規格分析（分析期）：** AI Agent 分析視圖層切分，**僅產出**實作 UI 元件的 spec/task（不進行任何實作）。
2.  **UI 元件開發（實作期）：** 根據前一步的 spec/task，使用假資料進行 UI 元件開發。

### 階段三：API 請求對接
3.  **API 請求實作：** 實作 API 規格、參數收集與請求發送，確認能順利取回後端最原始的資料結構。

### 階段四：整合與造橋（延後分析與實作）
4.  **轉譯規則分析（分析期 - 延後執行）：** 當 UI 元件（Mock 驗證過）與 API 原始資料皆就緒後，AI Agent 比對兩端資料結構的落差，**僅產出**資料正規化/轉譯的 spec/task。
5.  **轉譯邏輯實作（實作期）：** 根據轉譯 spec/task，實作資料正規化層，將 API 原始資料轉換並注入到呈現層，完成系統縫合。

### 階段五：狀態連動與邊界控制
6.  **狀態與控制實作：** 實作純視覺開關與全域事件偵測的狀態連動。

---

## 3. 延後「資料正規化」分析與實作的優勢

*   **避免無效假設與重工：** 在開發初期，UI 與 API 的規格仍可能有變動。若過早進行轉譯邏輯的分析與設計，極易因為規格調整而需要重複修改。
*   **實現雙軌平行開發：** UI 呈現與 API 請求的開發可以完全獨立且並行，互不阻塞，直到最後整合期才透過正規化層將兩者縫合。
*   **職責解耦：** 確保 UI 元件只處理純粹的渲染與假資料，API 只處理純粹的資料取得，將變動最頻繁的轉譯邏輯收攏在正規化層。
